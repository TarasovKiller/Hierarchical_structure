Перспектива оптимизации

Создание партиций

1. Одна из перспектив оптимизации - это разделение таблицы на три части по столбцу type

CREATE TABLE offices (
    id SERIAL,
	  parent_id INTEGER,
    name VARCHAR(255),
    type INTEGER
   
)
PARTITION BY LIST (type);

CREATE TABLE office_part PARTITION OF offices
    FOR VALUES IN (1);

CREATE TABLE department_part PARTITION OF offices
    FOR VALUES IN (2);

CREATE TABLE employee_part PARTITION OF offices
    FOR VALUES IN (3);

2. Создание индексов для них:

CREATE INDEX office_part_type_parent_id_idx ON office_part (parent_id);
CREATE INDEX department_part_type_parent_id_idx ON department_part (parent_id);
CREATE INDEX employee_part_type_parent_id_idx ON employee_part (parent_id);

CREATE INDEX office_part_type_id_idx ON office_part (id);
CREATE INDEX department_part_type_id_idx ON department_part (id);
CREATE INDEX employee_part_type_id_idx ON employee_part (id);

  1. SQL-запрос
WITH staff_ancestor as (
	SELECT parent_id FROM offices
	WHERE type=3 and id=<id сотрудника>),
	department_ancestor as (
		SELECT * FROM offices
		WHERE type=2 and id = (SELECT parent_id FROM staff_ancestor)),
		office_ancestor as (
			SELECT id FROM offices
			WHERE type=1 and id = (SELECT parent_id FROM department_ancestor)),
			department_descendant AS (
				SELECT id FROM offices
				WHERE type=2 and parent_id=(SELECT id FROM office_ancestor)),
				staff_descendant as (
					SELECT name FROM offices
					WHERE type=3 and parent_id in (SELECT id FROM department_descendant))

SELECT name from staff_descendant

Добавление столбца path

  1. Таблица будет выглядеть примерно так
id | name         | parent_id|path
---|--------------|----------|-----
1  | Офис в Москве| null     | 1
2  | Отдел продаж | 1        | 1/2
3  | Дима         | 2        | 1/2/3
4  | Костя        | 2        | 1/2/4
5  | Паша         | 2        | 1/2/5
  1. SQL запрос упрощается и становится более читабельным. 
WITH ancestor AS (
		SELECT split_part(path, '/', 1) AS p
		FROM offices
		WHERE id = 100000
	)
	SELECT * FROM offices WHERE path LIKE (SELECT p from ancestor)||'/%';
  1. Не забываем использовать индексы: здесь я решил воспользоваться GIN. Он поддерживает поиск по шаблону LIKE с использованием оператора @@. Он может быть эффективным для запросов, которые используют LIKE с шаблоном, начинающимся с конкретного значения, как в нашем случае.
CREATE EXTENSION pg_trgm;
CREATE INDEX idx_offices_path ON offices USING gin (path gin_trgm_ops);
